Java Multi thread

Posted on 2017-05-13

第一章

  1. 线程安全问题会出现在实例变量上，方法内部私有变量不会出现线程安全问题
  2. i--;的实际过程分为3个步骤，所以在多个线程执行这一句语句也会出现非线程安全
  3. System.out.println(i--);的线程安全问题：println()是sync方法，但是i--可能在进入方法之前先执行，此语句也是非线程安全的
  4. 如果将Thread对象以构造参数的方式传给一个Thread对象，Thread.currentThread()和this指的不是同一个线程对象
  5. sleep(long)是使当前正在执行的线程(this.currentThread()返回的)进入休眠
  6. 停止线程的方法：使用interrupt()给线程打一个停止标记，自行判断该标记通过抛异常方式停止线程执行
  7. 判断线程的停止标记
    1. thread.interrupted()：static方法，测试当前线程是否已中断，”当前线程”指运行这个语句的线程，而不是thread表示的线程，此方法会清除线程中断标志
    2. thread.isInterrupted()：测试thread是否已中断，不会清除线程中断标志
  8. 在sleep()状态下停止线程，会抛出异常，并会清除线程中断标志
  9. 线程优先级具有继承性：如果A线程启动了B线程，那么B线程会继承A线程的优先级
  10. 守护线程：当进程中不存在非守护线程时，守护线程会自动销毁。setDaemon(true)设置为守护线程

第二章

  1. 使用synchronized修饰普通方法时，获取的锁是当前对象锁，当某一线程持有了该对象锁，其它线程想要执行需要此对象锁的sync方法都会被阻塞
  2. synchronized的锁可以是任意对象，持同一对象的sync代码是同步执行的
  3. 使用synchronized修饰static方法时，表示对当前*.java文件对应的class持锁，和对象锁是不同的
  4. 进入sleep的线程不会释放锁，出现异常时线程会释放锁
  5. synchronized的锁重入：
    1. 当一个线程得到锁后，再次请求是一定会得到的，即可以在一个sync方法中调用另一个sync方法
    2. 子类可通过此特性立即获得父类锁(父类锁没被占有的情况下)，调用父类同步方法
  6. 同步方法不会被继承，子类重写父类sync方法时，不加sync则失去同步效果，加sync才能同步

  7. 数据类型String具有常量池缓存功能

    String a = "a";
String b = "a";
a == b;   // true

    因此大多数情况下应避免使用String作为锁对象

  8. 对象的属性改变并不影响锁：例如原来lock=”123”，后来再赋值lock=”456”，还是同一个锁对象
  9. volatile关键字用于修饰变量，使变量在多个线程间可见。每个线程都会有一个自己的私有栈存放变量，写入和读取都在私有栈上可提高效率，但会出现值修改后线程间不同步的现象，volatile强制线程从主内存中读写变量，使变量的更新对其它线程可见
  10. 多线程访问volatile不会阻塞，volatile只保证数据的可见性，但不保证原子性，可能会出现非线程安全
  11. 将变量作为参数传给synchronized也可以实现可见性
  12. volatile与synchronized的区别：
    1. volatile解决的是变量在多个线程之间的可见性
    2. synchronized解决的是多个线程之间访问资源的同步性

第三章

  1. wait/notify机制
    1. wait/notify/notifyAll都是在sync代码块中使用的
    2. 调用wait()之前线程要获得该对象的对象级别的锁，调用notify时持相同的锁
    3. notify()调用后并不是立即释放锁，只有等到执行完notify()所在的sync代码块后才会释放锁
  2. join()方法内部使用的是wait()实现，与sleep不同，join会释放锁
  3. ThreadLocal类用于给每个对象绑定自己的私有数据，解决了变量在不同线程之间的隔离性

第四章

  • java.util.concurrent.locks.Lock
  • java.util.concurrent.locks.ReentrantLock

  • java.util.concurrent.locks.Condition

    Lock lock = new ReentrantLock();
当线程执行了lock.lock()后，在执行lock.unlock()之前，中间的程序是加锁的
Condition c = lock.newCondition();
当线程执行了lock.lock()获得锁后，便可调用c.await()使线程进入WAITING状态，并释放锁
使用另一线程获得锁后，执行c.signal()将被c对象await的线程唤醒

使用多个Condition对象，则调用某个c.signal()/signalAll()只会唤醒调用了c.await()的线程

  • Lock类分为公平锁和非公平锁

    Lock lock = new ReentrantLock(Boolean );
true表示公平锁，线程获得锁的顺序按照线程来到lock.lock()的顺序
false表示非公平锁，则线程按抢占式获得锁

  • 读写锁ReentrantReadWriteLock:

    Lock lock = new ReentrantReadWriteLock();
lock.readLock().lock();
lock.writeLock().lock();

    读锁是共享锁，可多线程获得。
    写锁是排他锁，获得此锁后其它线程在此锁释放前不可以获读、写锁

第五章

  1. java.util.Timer 用于设置计划任务，负责在指定的时间执行某个任务

  2. java.util.TimerTask 封装任务的内容。抽象类，implements Runnable.

    timer.schedule(TimerTask task, Date time)
//Date若晚于当前时间，则到时间执行，若早于当前时间则立即执行
//一个Timer可以schedule多个TimerTask，Task会按设置顺序依次执行

timer.schedule(TimerTask task, Date firstTime, long period)
//在指定日期后，按指定周期循环执行
  3. task.cancel() 将自身从任务队列中清除；timer.cancel() 清除任务队列

  4. timer.scheduleAtFixedRate(TimerTask task, Date firstTime, long period)

    与schedule的区别：schedule()没有追赶性，此方法没有追赶性

追赶性：计划执行时间早于当前时间，那么少掉的那些任务项scheduleAtFixedRate()会立即开始全部执行回来
